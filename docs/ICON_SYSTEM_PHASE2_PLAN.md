# 图标系统二期需求与实施方案总结

## 1. 背景

本次计划要做的是导航站点图标系统二期，目标不是只“抓到图标”，而是把图标作为一套可管理的资源来处理，解决以下实际问题：

- 源站图标经常加载慢、不稳定，甚至直接失效
- 现有站点很多，需要支持批量补齐和统一迁移
- 相同域名下很多链接其实共用同一个图标，不应该重复下载、重复存储、重复上传
- 当源站没有图标或抓取失败时，系统需要有稳定兜底方案
- 后台需要可视化管理，而不是只有零散接口

本轮尝试最终没有保留为正式实现，原因是改动范围过大，影响了原有项目稳定性。但需求、方案和实施经验是有价值的，建议下次基于 `main` 干净分支重新实施。

---

## 2. 最终确认过的需求

### 2.1 后台功能模块

- 需要有独立的“图标管理”后台模块
- 旧的“数据管理 -> 抓取缺失图标”入口不能直接消失
- 数据管理页里要保留一个图标任务入口，否则页面观感会像“缺一块”
- 完整配置和批量任务应统一收口到“图标管理”模块

### 2.2 图标显示来源切换

系统需要支持三种图标来源：

- 源站图标
- 本地图标
- 图床图标

并且支持：

- 全局默认显示模式
- 单站点覆盖显示模式
- 推荐增加 `smart` 模式

建议优先级：

- `smart`: 本地 -> 图床 -> 源站

### 2.3 图标同步能力

需要支持：

- 批量把已有站点图标下载到本地
- 批量把已有站点图标上传到图床
- 新增站点时自动抓取源站图标
- 单站点可选择是否同步到本地
- 单站点可选择是否同步到图床
- 后续新增站点时，沿用全局默认策略，但允许单站点覆盖

### 2.4 抓取失败兜底

必须考虑以下情况：

- 获取源站图标失败
- 源站本身没有图标
- 图标 URL 失效
- 图床上传失败

需要的兜底能力：

- 支持手动填写图标 URL
- 支持手动上传图标文件
- 手动上传的图标优先级最高

### 2.5 去重与复用

这是本期非常关键的需求。

对于相同域名不同链接，例如 GitHub 项目页：

- 不应该每个链接都单独下载图标
- 不应该每个链接都单独保存一份本地文件
- 不应该每个链接都单独上传一份图床资源

应该实现：

- 同域名图标复用
- 同源 URL 复用
- 同文件内容复用

## 3. 已确认的接入决策

### 3.1 图床

本期先接入：

- `EasyImage`

接口确认结果：

- 上传方式：`POST multipart/form-data`
- 关键字段：`token`、`image`
- 成功响应中包含 `url`

### 3.2 配置方式

图床配置不应写死在代码里，应放到后台设置中：

- provider
- api url
- token

原因：

- 更安全
- 可替换图床实例
- 后续可扩展到其他图床实现

### 3.3 旧入口保留策略

不保留旧的线程抓取实现，但保留旧接口壳层：

- `/api/batch-fetch-icons`
- `/api/batch-fetch-icons/status`
- `/api/batch-fetch-icons/stop`

内部改为调用新的图标任务系统。

## 4. 推荐的数据结构设计

建议采用“两层模型”：

### 4.1 `WebsiteIcon`

表示“某个站点当前的图标策略和状态”。

建议字段：

- `website_id`
- `icon_asset_id`
- `domain_key`
- `source_mode`
- `display_mode_override`
- `sync_local_mode`
- `sync_imagebed_mode`
- `fetch_status`
- `local_status`
- `imagebed_status`
- `last_fetch_at`
- `last_local_sync_at`
- `last_imagebed_sync_at`
- `last_error`

说明：

- `WebsiteIcon` 不负责真正存文件
- 它只负责描述站点图标状态和策略

### 4.2 `IconAsset`

表示“可复用的图标资源实体”。

建议字段：

- `domain_key`
- `file_hash`
- `source_url`
- `source_host`
- `local_path`
- `mime_type`
- `imagebed_provider`
- `imagebed_url`
- `imagebed_delete_url`
- `imagebed_payload_json`

说明：

- 同一个图标资源可以被多个网站共用
- 复用的核心对象应该是 `IconAsset`

### 4.3 `SiteSettings`

图标相关全局设置建议增加：

- `icon_display_mode`
- `icon_auto_fetch_on_create`
- `icon_default_sync_local`
- `icon_default_sync_imagebed`
- `icon_imagebed_provider`
- `icon_imagebed_api_url`
- `icon_imagebed_token`

## 5. 推荐的后端逻辑设计

### 5.1 图标来源模式

建议支持：

- `auto`
- `manual_url`
- `manual_upload`

说明：

- `auto`：系统自动抓取
- `manual_url`：管理员手动填写 URL
- `manual_upload`：管理员手动上传文件

### 5.2 显示模式

建议支持：

- `inherit`
- `source`
- `local`
- `imagebed`

全局设置默认值建议支持：

- `smart`
- `source`
- `local`
- `imagebed`

### 5.3 复用策略

建议复用顺序：

1. 先按 `source_url` 精确复用
2. 再按 `domain_key` 复用同域名已有资产
3. 最终按文件内容 `sha256` 去重

### 5.4 本地存储策略

不建议继续按站点一站一目录存放。

建议改成共享存储：

- `static/uploads/site_icons/assets/<hash-prefix>/<hash>.<ext>`

优点：

- 便于复用
- 便于去重
- 删除站点时不会误删其他站点正在使用的图标

### 5.5 图床上传策略

推荐流程：

1. 若已有共享资产且已上传图床，直接复用
2. 若没有本地文件，先下载或使用手动上传文件落本地
3. 再上传图床
4. 回填图床 URL 和返回 payload

### 5.6 删除策略

删除网站时不应简单删文件夹。

正确逻辑应是：

1. 解除 `WebsiteIcon -> IconAsset` 关联
2. 检查该 `IconAsset` 是否还有其他网站引用
3. 若没有引用，才删除本地文件和 `IconAsset` 记录

## 6. 推荐的前端与后台页面设计

### 6.1 图标管理页

建议包含四块：

1. 概览统计
2. 全局设置
3. 批量任务
4. 问题站点列表

概览统计建议显示：

- 站点总数
- 源站图标可用数
- 本地图标可用数
- 图床图标可用数
- 手动上传数
- 共享资产数
- 抓取失败数
- 本地失败数
- 图床失败数

全局设置建议显示：

- 默认显示模式
- 新建站点时自动抓取
- 默认同步到本地
- 默认同步到图床
- 图床 provider
- 图床 API URL
- 图床 token

批量任务建议提供按钮：

- 补齐缺失图标
- 同步到本地
- 同步到图床
- 重试失败项

### 6.2 站点编辑页

建议新增：

- 显示模式选择
- 同步到本地 checkbox
- 同步到图床 checkbox
- 手动上传图标 input
- 当前图标状态展示

状态展示建议包含：

- 当前最终显示图标
- 来源 origin
- `fetch_status`
- `local_status`
- `imagebed_status`
- 是否共享资产
- 最后错误信息

### 6.3 数据管理页

建议保留“抓取缺失图标”卡片，但说明文案改成：

- 这是图标任务快捷入口
- 完整配置和更多任务请去图标管理页

并增加：

- “进入图标管理”按钮

## 7. 使用逻辑设计

### 7.1 新增站点

建议流程：

1. 管理员新增站点
2. 若未填写图标 URL 且开启自动抓取，则抓取源站图标
3. 根据全局默认或单站点配置决定是否同步到本地
4. 根据全局默认或单站点配置决定是否同步到图床
5. 保存 `WebsiteIcon` 状态

### 7.2 已有站点批量迁移

建议先做两类任务：

- 同步到本地
- 同步到图床

并支持：

- 只处理缺失项
- 失败项重试

### 7.3 源站失败时

建议回退顺序：

1. 如果有本地图标，显示本地
2. 如果有图床图标，显示图床
3. 如果有手动 URL，显示手动 URL
4. 如果管理员上传过图标，优先显示上传图标
5. 否则显示默认占位图

## 8. 本轮失败的主要原因

这次不是需求有问题，而是实施方式有问题。

### 8.1 改动范围过大

一次性同时改了：

- 模型
- 数据迁移
- 图标服务
- 后台路由
- 多个模板
- 旧接口兼容层

导致影响面过宽，回归成本太高。

### 8.2 仓库本身有历史负担

包括：

- 部分文件编码混乱
- 后台模块已有历史脏代码
- 原项目在 Windows `\\?\\` 路径下本身就有路径问题

这会把本来应该聚焦在“图标功能”的修改，扩散成“系统级兼容修复”。

### 8.3 缺少分阶段验收

更合理的做法应该是：

1. 先只做模型和迁移
2. 再做服务层
3. 再做后台页
4. 再做旧入口兼容
5. 每一步都真实启动项目验收

## 9. 下次实施建议

### 9.1 必须从干净的 `main` 新开分支

不要在带大量历史改动的分支上继续堆功能。

### 9.2 分四次提交

建议拆成：

1. 数据结构与迁移
2. 图标服务与共享资产
3. 后台图标管理与站点编辑页
4. 旧入口兼容与批量任务

### 9.3 每一步都要做真启动验证

至少验证：

- `python run.py` 能启动
- `/` 正常
- `/admin/websites` 正常
- `/admin/icon-management` 正常
- 编辑站点时表单能保存
- 批量任务接口能返回

### 9.4 先保守，再增强

建议首次落地先做：

- 本地图标共享资产
- 手动上传
- 显示模式切换
- EasyImage 接入

可以暂缓的增强项：

- 删除图床资源
- 更丰富的图床适配
- 更复杂的统计面板

---

## 10. 推荐的实施顺序

建议严格按顺序做：

1. 建立 `IconAsset`、扩展 `WebsiteIcon`、扩展 `SiteSettings`
2. 做数据库迁移和历史数据回填
3. 重写图标服务层
4. 接入单站点编辑页
5. 接入图标管理页
6. 接入旧数据管理页快捷入口
7. 接入旧 `icon_fetch` 兼容接口
8. 做真实 EasyImage 上传测试
9. 做真实批量同步测试
10. 最后清理旧文件和无用入口

---

## 11. 验收标准

下次重新实施时，至少要满足以下验收条件：

- 项目可正常启动
- 首页可正常打开
- 后台可正常登录
- 图标管理页可打开
- 网站编辑页可设置图标策略
- 手动上传图标可成功显示
- 本地图标可批量同步
- 图床可批量同步
- 同域名图标能复用，不重复保存
- 删除网站后不会误删其他网站共用图标
- 数据管理页保留图标任务入口

---

## 12. 一句话结论

这次实现虽然没有保留，但需求方向是正确的。下次应基于干净 `main`，按“数据结构 -> 服务层 -> 后台页面 -> 兼容入口 -> 验收”的顺序重做，避免一次性大改破坏原有功能。
